## Техническое задание: API для персонального финансового контроля

### 1) Легенда

Вы разрабатываете учебное API для персонального финансового контроля. Пользователи могут добавлять доходы и расходы, устанавливать лимиты по категориям и получать рекомендации по экономии.

Все данные хранятся в памяти — без базы данных и внешних зависимостей. (Но подготовить код к использованию БД)  
Система должна быть **расширяемой**: в будущем могут появиться новые типы рекомендаций и способы анализа, и добавление их не должно требовать изменения основной логики анализа  (Open/Close Principe).

---

### 2) Общие правила

- `userId` и `id` транзакций — **целые числа (`int`)**.  
- Все суммы — `decimal` (например, `1250.75`).  
- Категории — непустые строки (например, `"Продукты"`, `"Зарплата"`).  
- Доходы — увеличивают `amount`.  
- Расходы — уменьшают `amount`.  
- При анализе бюджета и рекомендациях **учитываются только расходы** (модуль отрицательных сумм).

---

### 3) Логика рекомендаций по экономии

Рекомендации генерируются **только для категорий с установленным бюджетом** и **только если**:

> **Фактические расходы ≥ 80% от лимита**

В этом случае возвращается одна рекомендация на такую категорию вида:  
> *«Вы использовали {процент}% бюджета в категории "{Категория}". Рассмотрите возможность сократить расходы.»*

- `процент` = `Math.Round((actualSpent / budgetLimit) * 100)`
- `potentialSavings` = `0` (на данном этапе не рассчитывается реальная экономия)
- Если таких категорий несколько — возвращаются все.
- Если ни одна категория не проходит порог в 80% — возвращается пустой массив.

> Эта логика должна быть реализована так, чтобы в будущем можно было легко добавить другие правила (например, анализ регулярных подписок) без изменения основного кода анализа.

---

### 4) Эндпоинты

#### `POST /transactions`

**Тело запроса**:
```json
{
  "userId": "int",
  "description": "string, не пустой",
  "amount": "decimal, ≠ 0",
  "category": "string, не пустой"
}
```

**Успешный ответ (201 Created)**:
```json
{
  "id": "int",
  "userId": "int",
  "description": "string",
  "amount": "decimal",
  "category": "string"
}
```

**Ошибки**: `400 Bad Request` при нарушении валидации.

---

#### `GET /transactions?userId={int}`

**Успешный ответ (200 OK)**: массив всех транзакций пользователя.

---

#### `PUT /budgets`

**Тело запроса**:
```json
{
  "userId": "int",
  "category": "string, не пустой",
  "limit": "decimal, > 0"
}
```

**Успешный ответ (200 OK)**:
```json
{
  "userId": "int",
  "category": "string",
  "limit": "decimal"
}
```

---

#### `GET /budgets/analysis?userId={int}`

**Успешный ответ (200 OK)**:
```json
[
  {
    "category": "string",
    "budgetLimit": "decimal",
    "actualSpent": "decimal (≥ 0)",
    "isOverBudget": "boolean",
    "remaining": "decimal"
  }
]
```

> Включать все категории, где есть **бюджет или хотя бы одна транзакция**.

---

#### `GET /insights/savings?userId={int}`

**Успешный ответ (200 OK)**:
```json
[
  {
    "message": "string",
    "potentialSavings": 0,
    "category": "string"
  }
]
```


### 5) Требования к тестированию

Написать юнит-тесты для следующих сценариев:

1. **Валидации транзакции**:  
   - amount = 0 → ошибка  
   - пустая категория → ошибка  

2. **Анализ бюджета**:  
   - Лимит = 200, расходы = 500 → isOverBudget = true, remaining = -300

3. **Рекомендации**:  
   - Лимит = 1000, расходы = 950 → возвращается рекомендация с текстом, содержащим «95%» и «Кино»  
   - Лимит = 1000, расходы = 700 → не возвращается (менее 80%)

Данные юнит-тесты являются лишь примерами, какие ещё функции протестировать – решай сам.
